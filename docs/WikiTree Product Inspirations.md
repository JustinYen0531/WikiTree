# WikiTree Product Inspirations

這份文件用來整理 WikiTree 借鑑了哪些產品、各自借了什麼、哪些是核心靈感、哪些只是實作手段。

它的目的不是證明 WikiTree 像誰，而是幫助回溯：

- 某個功能的源頭來自哪種產品思路
- 哪些借鑑是產品哲學
- 哪些借鑑只是操作層、編輯層或技術層
- WikiTree 最後如何把這些想法拼成自己的東西

## 1. 總結：WikiTree 不是某個產品的複製品

WikiTree 本質上是一個混合體。

它不是單純借鑑一個產品，而是把不同產品各自擅長的部分拆開，再重新組成一個新的知識工作流。

最接近的說法不是：

> WikiTree 像某某產品。

而是：

> WikiTree 借用了多種成熟產品的局部精神，但目的是建立一個校園型、社群型、知識版本控制平台。

因此，這些產品對 WikiTree 的影響可以分成四層：

- 產品骨架
- 編輯與儲存方式
- 協作與社群機制
- 探索、推薦與獎勵

## 2. 第一層：產品骨架

### GitHub

GitHub 是 WikiTree 最深層的骨架來源。

WikiTree 從 GitHub 借來的，不是寫程式，而是：

- Fork
- Review
- Pull Request
- Merge
- 版本觀念
- Credit 與貢獻者可見性

GitHub 解決的是：

> 陌生人如何安全地共同修改同一個作品。

WikiTree 想把這件事翻譯成知識世界：

> 同學、學長姐、老師、陌生人，如何安全地共同培育同一棵知識樹。

如果要問 WikiTree 最核心的 DNA 來自哪裡，答案是 GitHub。

### Wikipedia

Wikipedia 提供的是「知識應該公開、可被共同維護」這個觀念。

WikiTree 從 Wikipedia 借來的不是百科文章格式，而是：

- 知識開放
- 公共知識庫
- 社群共同維護
- 一個知識點可以成為公共入口

但 WikiTree 和 Wikipedia 的差別很明顯：

- Wikipedia 偏向靜態百科
- WikiTree 偏向可分支、可 Fork、可迭代的知識筆記

所以 Wikipedia 更像是知識公共性的來源，而不是產品操作方式的來源。

## 3. 第二層：編輯與儲存方式

### Obsidian

Obsidian 提供的是本地 Markdown 筆記管理的觀念。

WikiTree 從 Obsidian 借來的部分包括：

- 筆記本質是本地檔案
- 使用者打開的是自己真正擁有的 Markdown 文件
- 不需要一開始就把所有內容塞進雲端資料庫
- 本地創作與本地整理可以先成立

這一點很重要，因為 WikiTree 並不是一開始就想當純雲端資料庫產品。

它保留了：

> 我的創作可以先是我的本地文件。

而線上探索與社群內容，則是另一個層次。

### Notion

Notion 提供的是「線上同步與雲端狀態」的感覺，但 WikiTree 沒有照抄 Notion 的同步哲學。

WikiTree 借來的是：

- 線上可存取
- 跨裝置可使用
- 雲端狀態存在
- 發布後成為網路上的可見版本

但 WikiTree 和 Notion 的最大差異是：

- Notion 比較像持續同步的工作區
- WikiTree 比較像由使用者決定何時把一個版本公開到網路上

也就是說，WikiTree 的雲端不是「我每打一次字都同步」，而是：

> 我完成一個我願意公開的版本，再把它發布出去。

這使它比較像「發版」而不是「每秒同步」。

### Typora

Typora 提供的是 Enough Editor 的審美與手感標竿。

WikiTree 從 Typora 借來的是：

- 極簡
- 夠用
- 該有的 Markdown 語法都支援
- 編輯時不要讓工具感過重

Typora 對 WikiTree 的影響比較偏編輯體驗層，而不是產品核心定位。

如果 GitHub 是骨架，Typora 更像是手感標準。

### HackMD

HackMD 提供的是線上 Markdown 協作文件的參考系。

WikiTree 從 HackMD 借來的部分包括：

- 線上 Markdown 編輯
- 文件型協作感
- 輕量分享與閱讀

但 HackMD 對 WikiTree 比較像一個「邊界對照物」。

因為 HackMD 強調的是：

- 多人同時共編
- 即時同步
- 團隊共同完成一份文件

而 WikiTree 要的不是這個。

WikiTree 比較像：

- 幕後整理
- 做好再發布
- Fork 後各自長出版本
- 透過 Review / Merge 把知識帶回主樹

所以 HackMD 是一個「看起來像，但本質不同」的重要參考物。

## 4. 第三層：協作與社群機制

### Codex / AI Agent 類產品

Codex、AI sidebar、AI agent 類產品提供的是：

- 選取一段內容後協助改寫
- 在側邊欄提供上下文建議
- Review / Clarify / Expand / Fix
- 不打斷主要編輯流程

WikiTree 從這類產品借來的不是完整代理，而是：

- AI 應該長在知識裡
- AI 應該輔助修改，不該蓋過內容本身
- AI 最好是 contextual，而不是永遠只做聊天框

但由於瀏覽器安全限制，WikiTree 目前的 agent 比較像 semi-agent：

- 能選取
- 能建議
- 能回寫部分內容
- 不能像桌面代理一樣完整操作整個環境

所以這一層的借鑑是互動方式，而不是能力上限。

### AI 的中立底稿觀

WikiTree 還借用了另一種比較不屬於單一產品的觀念：

> 先由相對中立的知識整理形成共同底稿，再由人依照情境把它改成更有目的性的版本。

這個靈感並不完全來自某個具名軟體，而比較像來自近年生成式 AI 在學習場景中的一種普遍特性：

- 它傾向先給出較通用的整理
- 它不會天然知道某位老師最愛考什麼
- 它比較像共同範圍的整理器，而不是特定考場策略本身

WikiTree 借來的不是「AI 自動完成一切」，而是這個分層思路：

- 先有相對中立的共同底稿
- 再有人把它修成課程版
- 再有人把它修成老師版
- 再有人把它修成考試版

這讓 WikiTree 的筆記不必從空白頁開始，而可以從共同底稿開始演化。

### 校園社群 / 共同筆記文化

雖然這不是單一軟體，但它其實是 WikiTree 很重要的來源。

WikiTree 想承接的是一種校園裡本來就存在的行為：

- 學長姐留下筆記
- 同學互相交換整理
- 有人補例子
- 有人做考前版
- 有人加自己的理解

只是以前這些東西：

- 分散
- 匿名
- 難以追溯
- 沒有 Credit
- 無法形成版本血緣

WikiTree 想把這種本來就在校園中發生的知識流動，做成有結構的產品。

## 5. 第四層：探索、推薦與獎勵

### YouTube / 推薦系統產品

WikiTree 未來想做的 Knowledge Discovery，不是原創於 WikiTree。

它借用的是成熟推薦產品的邏輯：

- 看過這個的人，下一步常看什麼
- 你可能對哪個知識點產生興趣
- 不只推薦熱門，也能推薦少見但有趣的路徑

這種思路比較像 YouTube 或其他內容平台的推薦機制。

WikiTree 要借來的是：

> 不只是回答問題，而是把人帶進下一個知識坑。

### 遊戲化 / 成就系統

WikiTree 的聲望、澆水、分枝、回枝、傳承值，也不是完全原創。

它借來的是很多成熟產品共同使用的獎勵設計：

- 讚
- 分數
- 等級
- 成就
- 貢獻值
- 個人檔案可見數據

但 WikiTree 的差別是：這些數值不應該獎勵短期流量，而應該獎勵可追溯的知識貢獻。

所以 WikiTree 的 gamification 不是普通社群積分，而比較像：

- 知識影響力
- 貢獻傳承度
- 對知識森林的照顧程度

## 7. 哪些是核心，哪些不是

如果要再壓縮一次，可以這樣分：

### 最核心的三個來源

- GitHub：決定了 WikiTree 的骨架
- Wikipedia：決定了 WikiTree 的公共知識觀
- 校園共同筆記文化：決定了 WikiTree 為什麼成立

### 第二層的重要來源

- Obsidian：決定本地文件思路
- Notion：決定線上發布與可存取感
- Typora：決定 enough editor 的手感

### 第三層的支援來源

- Codex / AI Agent：決定 AI 如何介入編輯
- HackMD：決定你不要走成什麼樣
- 推薦系統產品：決定未來的探索引擎
- 遊戲化產品：決定貢獻誘因如何被設計

## 8. 最後的整理

如果要用一句話概括：

> WikiTree 是把 GitHub 的版本協作、Wikipedia 的公共知識觀、Obsidian 的本地文件思維、Typora 的簡潔編輯體驗、Notion 的線上可見性、AI agent 的輔助改寫方式、推薦系統的知識探索邏輯，以及校園共同筆記文化，重新拼成一個校園型知識版本平台。

所以 WikiTree 的重點不是「像很多產品」。

重點是：

> 它把很多成熟產品各自擅長的一塊，接成一條新的 Knowledge Workflow。
